Good Security Practices Identified

CRITICAL SECURITY ISSUES

1. JWT Verification Disabled on ALL Edge Functions HIGH SEVERITY

Your supabase/config.toml has disabled JWT verification on all 5 edge functions:

[functions.analyze-ux]
verify_jwt = false

[functions.create-razorpay-order]
verify_jwt = false

[functions.verify-razorpay-payment]  
verify_jwt = false

[functions.check-user-subscription]
verify_jwt = false

[functions.razorpay-webhook]
verify_jwt = false

Impact: This is extremely dangerous because:

  • Anyone on the internet can call these functions without authentication
  • Attackers can create unlimited subscriptions for any user ID
  • Attackers can manipulate subscription data (plan types, limits, usage counts)
  • Your OpenRouter API credits can be drained by unauthorized users
  • The analyze-ux function attempts to authenticate users, but with verify_jwt = false, this check is meaningless

Why this happened: Only the razorpay-webhook function legitimately needs verify_jwt = false (since webhooks come from Razorpay, not authenticated users). The other 4 functions should require JWT authentication.

2. User ID Tampering Vulnerability CRITICAL SEVERITY

In multiple edge functions, the userId is accepted directly from the request body without validation:

create-razorpay-order/index.ts:

const { planType, userId }: CreateOrderRequest = await req.json();
// userId is used directly without checking auth.uid()

verify-razorpay-payment/index.ts:

const { razorpay_payment_id, razorpay_order_id, razorpay_signature, userId } = await req.json();
// userId is used directly to update subscriptions

check-user-subscription/index.ts:

const { userId }: CheckSubscriptionRequest = await req.json();
// userId is used directly to fetch/create subscriptions

Impact:

  • An attacker can pass any user ID in the request
  • They can upgrade other users' plans without payment
  • They can check or manipulate any user's subscription data
  • Combined with disabled JWT verification, this allows complete account takeover

Correct approach: Extract the user ID from the JWT token using auth.uid(), never trust client-provided user IDs.

3. Race Condition in Payment Verification MEDIUM SEVERITY

The payment flow has a potential race condition:

  1. create-razorpay-order stores razorpay_order_id and sets payment_status: 'pending'
  2. User completes payment on Razorpay
  3. TWO parallel processes may update the subscription:
    • Frontend calls verify-razorpay-payment
    • Razorpay sends webhook to razorpay-webhook

Current data shows this issue:

Impact:

  • Duplicate subscription updates
  • Inconsistent subscription state
  • Potential for users to be charged but not activated

4. Missing Input Validation MEDIUM SEVERITY

analyze-ux function:

  • The url parameter is passed directly to the AI without validation
  • No URL format validation, length limits, or domain restrictions
  • Potential for injection attacks or abuse

Frontend validation only:

  • All edge functions rely on client-side validation which can be bypassed
  • No server-side schema validation using libraries like Zod

5. Hardcoded Clerk Publishable Key LOW SEVERITY

In src/main.tsx:

const PUBLISHABLE_KEY = "pk_test_ZGVzaXJlZC1mb3gtNTQuY2xlcmsuYWNjb3VudHMuZGV2JA";

While publishable keys are designed to be public, this is a test key (indicated by pk_test_).

The .env file contains a live Razorpay key ID. While this is a publishable key (safe to expose), it's visible in your codebase.
